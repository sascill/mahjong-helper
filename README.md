# Mahjong Helper

리치마작 초보자가 첫 손패를 받고 이번 판에 노려볼 만한 역을 빠르게 선택할 수 있도록 돕는 모바일 중심 PWA입니다.

## 프로젝트 목적

마작 초보자는 첫 손패를 받은 뒤 어떤 역을 목표로 패를 만들어야 할지 판단하기 어렵습니다.

패를 받을 때마다 역 목록을 다시 확인하면 게임의 흐름이 끊기고, 게임을 즐기기보다 규칙을 공부하는 시간이 길어집니다.

Mahjong Helper는 첫 손패를 한 번 입력하면 현재 손패에서 가까운 역을 제시하여, 플레이어가 이번 판의 목표를 빠르게 정할 수 있도록 돕습니다.

이 프로젝트는 플레이어 대신 패를 선택해 주는 AI가 아닙니다.

> 앱은 목표를 제시하고, 실제 조패와 판단은 플레이어가 직접 한다.

## 핵심 사용 흐름

```text
첫 손패 13장 선택
→ 장풍과 자풍 선택
→ 가까운 역 최대 3개 확인
→ 목표로 삼을 역 선택
→ 역의 조건과 완성 형태 확인
→ 휴대폰을 내려놓고 직접 플레이
```

## 추천 기준

각 역이 성립하는 완성패와 현재 손패를 비교하여, 화료까지 최소 몇 장의 패가 필요한지 계산합니다.

- 1장 필요: 이미 텐파이인 상태
- 2장 필요: 매우 가까운 목표
- 3장 필요: 가까운 목표
- 4장 필요: 노려볼 수 있는 목표
- 5장 필요: 다소 먼 목표
- 6장 이상 필요: 추천 대상에서 제외

가까운 순서대로 최대 3개의 역을 추천합니다.

이 값은 실제 화료 확률이나 앞으로 필요한 순서를 의미하지 않습니다. 가장 이상적으로 패를 진행했을 때 목표 역 완성까지 필요한 최소 패의 수입니다.

## MVP 기능

### 첫 손패 입력

- 만수·통수·삭수·자패 34종 제공
- 유니코드 마작패 사용
- 첫 손패 13장 선택
- 같은 패 최대 4장
- 선택 취소 및 전체 초기화
- 손패 자동 정렬
- 장풍 선택
- 자풍 선택

### 역 추천

- 현재 손패에서 가까운 역 계산
- 가까운 역 최대 3개 추천
- 완성까지 필요한 최소 장수 표시
- 추천 이유 제공
- 사용자가 추천된 역 중 목표를 직접 선택

### 역 도감

- 역 목록 탐색
- 역 이름과 판수
- 간단한 설명
- 성립 조건
- 멘젠 필수 여부
- 치·퐁·깡 가능 여부
- 유니코드 패로 구성한 완성 예시

## 예상 화면

1. 홈
   - 첫 손패 분석
   - 역 찾아보기
2. 손패 선택
   - 손패 13장 선택
   - 장풍·자풍 선택
   - 분석 실행
3. 추천 결과
   - 입력한 손패
   - 추천 역 최대 3개
   - 완성까지 필요한 최소 장수
   - 추천 이유
4. 역 목록
   - 지원하는 역 탐색
5. 역 상세
   - 판수, 조건, 제한 사항 및 완성 예시

## MVP 제외 범위

다음 기능은 초기 버전에 포함하지 않습니다.

- 사진 및 카메라 인식
- AI 모델 학습
- 버릴 패 추천
- 매 순서 실시간 조패 안내
- 실제 화료 확률 계산
- 후리텐 판정
- 도라 계산
- 부수 및 정확한 점수 계산
- 점수 관리
- 게임 기록 및 통계
- 세부 룰 설정
- 온라인 대전

## 초기 추천 대상 역

추천 알고리즘은 우선 다음 역부터 검증합니다.

- 탕야오
- 치또이츠
- 혼일색

거리 계산 방식이 검증되면 다음 역을 순차적으로 추가합니다.

- 핑후
- 역패
- 또이또이
- 일기통관
- 삼색동순
- 청일색
- 찬타
- 준찬타

리치와 멘젠쯔모처럼 첫 손패의 모양만으로 목표를 정하기 어려운 역은 메인 추천과 별도로 다룹니다.

## 기술 방향

현재 예정된 기술 구성입니다.

- React
- TypeScript
- Vite
- PWA
- 모바일 우선 반응형 UI
- 유니코드 마작패
- 별도의 서버 없이 동작하는 클라이언트 중심 구조

패 데이터와 화면 표시 방식을 분리하여, 이후 유니코드 패를 실제 이미지 에셋으로 교체할 수 있도록 구성합니다.

사진 인식 기능이 추가되더라도 추천 엔진은 변경하지 않고 다음과 같이 입력 방식만 확장합니다.

```text
직접 선택 → Tile[] → 추천 엔진

사진 촬영 → 패 인식 → Tile[] → 추천 엔진
```

## 개발 원칙

프로젝트는 다음 작업 순서를 따릅니다.

```text
스펙 작성
→ 테스트 코드 작성
→ 실제 구현
→ 코드 및 결과 리뷰
```

- 스펙 없이 기능을 구현하지 않습니다.
- 추천 알고리즘은 예상 결과가 정의된 테스트 손패를 먼저 작성합니다.
- 사용자에게 확률처럼 오해할 수 있는 임의 점수를 노출하지 않습니다.
- MVP 범위를 임의로 확장하지 않습니다.
- 초보자가 짧게 확인하고 다시 게임에 집중할 수 있는 경험을 우선합니다.

## 커밋 규칙

- `core:` 프로젝트 설정 및 핵심 기반 변경
- `feat:` 새로운 기능 추가
- `fix:` 오류 수정
- `docs:` 문서 추가 및 수정
- `refactor:` 기능 변경 없는 코드 구조 개선

## 문서 구성 예정

```text
docs/
├─ product-spec.md
├─ technical-spec.md
├─ recommendation-spec.md
├─ yaku-spec.md
└─ test-scenarios.md
```

- `product-spec.md`: 사용자 문제, 기능 범위, 화면 흐름
- `technical-spec.md`: 기술 구성과 프로젝트 구조
- `recommendation-spec.md`: 거리 계산과 추천 순위 규칙
- `yaku-spec.md`: 역별 성립 조건과 판정 기준
- `test-scenarios.md`: 검증용 손패와 기대 결과

## 프로젝트 상태

현재 프로젝트 초기 설정 및 스펙 작성 단계입니다.
