# AGENTS.md

## 적용 범위

이 파일의 규칙은 저장소 전체에 적용한다.

하위 디렉터리에 별도의 `AGENTS.md`가 있다면 해당 디렉터리에서는 더 구체적인 하위 규칙을 함께 적용한다.

사용자의 최신 지시가 이 문서와 충돌하면 사용자 지시를 우선한다.

## 문서 우선순위

작업을 시작하기 전에 해당 작업과 관련된 문서를 먼저 확인한다.

```text
docs/README.md
docs/product-spec.md
docs/technical-spec.md
작업 경로에서 가장 가까운 Spec.md
```

- 전체 문서와 모듈 경로는 `docs/README.md`를 따른다.
- 제품 기능과 범위는 `product-spec.md`를 따른다.
- 기술 구조와 개발 도구는 `technical-spec.md`를 따른다.
- 모듈의 인터페이스, 동작과 테스트 기준은 해당 구현 경로의 `Spec.md`를 따른다.
- 관련 스펙이 없거나 불명확하면 구현 전에 스펙부터 작성하거나 사용자에게 확인한다.
- 서로 다른 문서가 충돌하면 임의로 판단하지 않고 충돌 내용을 보고한다.

## 필수 작업 흐름

모든 기능 작업은 다음 순서를 따른다.

```text
스펙 작성 또는 확인
→ 테스트 작성
→ 테스트 실패 확인
→ 최소 구현
→ 전체 검증
→ 변경 사항 리뷰
```

### 1. 스펙

- 구현 전에 기능의 입력, 출력, 예외 조건과 완료 기준을 확정한다.
- 동작이 변경되면 관련 스펙을 먼저 수정한다.
- 스펙에 없는 기능을 임의로 추가하지 않는다.

### 2. 테스트

- 실제 구현보다 테스트를 먼저 작성한다.
- 버그 수정은 가능하면 문제를 재현하는 테스트부터 작성한다.
- 테스트가 올바른 이유로 실패하는지 확인한 뒤 구현한다.
- 테스트를 통과시키기 위해 검증을 약화하거나 테스트를 삭제하지 않는다.
- 테스트는 반복 실행해도 같은 결과가 나오도록 작성한다.

### 3. 구현

- 현재 스펙과 테스트를 만족하는 최소 범위만 구현한다.
- 요청과 관계없는 리팩터링이나 의존성 변경을 함께 수행하지 않는다.
- 기존 사용자 변경 사항을 덮어쓰거나 되돌리지 않는다.
- 임시 구현, 하드코딩과 가짜 데이터가 필요하면 명시하고 스펙에 허용된 범위에서만 사용한다.

### 4. 검증

작업 완료 전에 `docs/technical-spec.md`에 정의된 검증 명령을 실행한다.

최소 검증 대상은 다음과 같다.

- 관련 테스트
- 전체 테스트
- 타입 검사
- 린트
- 프로덕션 빌드

아직 존재하지 않는 명령을 임의로 가정하지 않는다. 프로젝트 설정이 완료되면 실제 명령을 이 문서 또는 `technical-spec.md`에 기록한다.

### 5. 리뷰

작업을 완료하기 전에 전체 변경 사항을 직접 검토한다.

- 구현이 스펙을 충족하는가?
- 테스트가 핵심 동작과 경계 조건을 검증하는가?
- MVP 범위를 벗어난 변경이 포함되지 않았는가?
- 불필요한 파일이나 로그가 포함되지 않았는가?
- 문서와 실제 동작이 일치하는가?
- 관련 없는 기존 코드가 변경되지 않았는가?

문서만 변경한 작업은 테스트를 생략할 수 있지만 문서 내용과 변경 diff는 검토한다.

## 범위 관리

- 새로운 기능을 임의로 추가하지 않는다.
- 요청과 직접 관계없는 개선은 별도 제안으로 남긴다.
- 카메라 인식, 점수 계산 등 MVP 제외 기능은 관련 스펙이 변경되기 전까지 구현하지 않는다.
- 추천 결과는 스펙에 정의된 기준만 사용하며 임의의 확률이나 점수를 사용자에게 노출하지 않는다.

## 문서 관리

- 프로젝트 전체에 적용되는 제품·기술 스펙은 `docs/`에서 관리한다.
- 도메인과 기능의 상세 스펙은 해당 구현 경로의 `Spec.md`에서 관리한다.
- 동일한 스펙을 전역 문서와 모듈 스펙에 중복해서 작성하지 않는다.
- 모듈을 추가·이동·삭제하면 `docs/README.md`의 설명과 경로를 함께 수정한다.
- `README.md`에는 상세 기술 스펙이나 작업 상태를 기록하지 않는다.
- 코드 동작이 변경되면 가장 가까운 `Spec.md`와 관련 테스트를 함께 갱신한다.
- 아직 확정되지 않은 내용을 확정된 요구사항처럼 작성하지 않는다.
- 문서 본문과 모든 코드 주석은 한국어로 작성한다.
- 코드 식별자와 파일명은 영어를 사용한다.

## Git 규칙

커밋은 하나의 목적을 가진 작은 단위로 작성한다.

허용하는 커밋 접두사는 다음과 같다.

- `core:` 프로젝트 설정 및 핵심 기반 변경
- `feat:` 새로운 기능 추가
- `fix:` 오류 수정
- `docs:` 문서 추가 및 수정
- `refactor:` 기능 변경 없는 코드 구조 개선

커밋 접두사는 위 형식을 유지하고, 접두사 뒤의 커밋 메시지는 한국어로 작성한다.

```text
docs: 에이전트 작업 규칙 정리
feat: 손패 선택 기능 추가
fix: 동일한 패를 다섯 장 선택할 수 있는 문제 수정
```

추가 규칙:

- 관련 없는 변경을 하나의 커밋에 섞지 않는다.
- 작업이 완료되면 사용자의 별도 요청을 기다리지 않고 즉시 커밋한다.
- 비밀 정보, 환경 변수와 개인 파일을 커밋하지 않는다.
- 생성 파일은 프로젝트에서 추적하도록 정한 경우에만 커밋한다.
- 기존 커밋을 임의로 수정하거나 강제 push하지 않는다.
- `main` 브랜치의 push는 사용자가 직접 수행한다.

## 브랜치와 PR

- 문서와 스펙만 변경하는 작업은 `main` 브랜치에서 진행할 수 있다.
- 테스트 코드나 실제 구현 코드를 작성하기 전에는 별도 작업 브랜치를 생성한다.
- 브랜치 이름은 커밋 유형과 작업 내용을 조합한 영어 이름을 사용한다.

```text
feat/hand-selector
fix/tile-count-limit
refactor/recommendation-engine
```

- 작업 브랜치에서는 테스트 작성, 구현, 검증과 자체 리뷰까지 완료한다.
- 작업이 끝나면 작업 브랜치를 원격 저장소에 push하고 PR을 생성한다.
- PR 제목과 설명은 한국어로 작성한다.
- PR 설명에는 관련 스펙, 주요 변경 사항, 검증 결과와 리뷰가 필요한 부분을 포함한다.
- PR을 생성한 뒤 직접 병합하지 않는다.
- PR 리뷰와 병합은 사용자 또는 별도의 리뷰 에이전트가 수행한다.

## 완료 조건

다음 조건을 모두 만족해야 작업을 완료한 것으로 판단한다.

- 관련 스펙이 작성되거나 최신 상태다.
- 필요한 테스트가 먼저 작성됐다.
- 구현이 스펙과 테스트를 충족한다.
- 정의된 검증 명령이 성공한다.
- 변경 사항을 직접 리뷰했다.
- 관련 없는 변경이 포함되지 않았다.
- 사용자에게 변경 내용과 검증 결과를 보고했다.
